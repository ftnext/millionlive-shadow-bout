# シャドウバウト v0.2 カード効果実装計画

Shadow Bout v0.1からv0.2へのアップデートとして、カードの「効果」を加味したシステムを実装します。この機能追加はドメインモデルからUIまで多岐にわたる変更が必要です。

## 実装方針（ユーザー合意事項）

前回の確認事項に基づき、以下の仕様で実装を進めます。

1. **効果の発動条件について**:
   「勝った場合」「負けた場合」などの効果（例：エレナ、このみ）は、「あいこになり効果処理が行われた後、ポイント勝負等を経て最終的に勝った/負けた場合、**次の勝負で**効果が発動する（遅延発動）」という仕様で実装します。
2. **実装範囲**:
   デッキに指定されている13枚のカード（百合子、杏奈、未来、亜美、ロコ、静香、千鶴、千早、貴音、真、瑞希、ジュリア、やよい）の**全ての効果**を実装します。
3. **NPCの挙動**:
   プレイヤーが選択を行う効果（例：百合子の「どちらかの効果を選んで発動する」等）について、NPCがそれを使用する際は**ランダムに選択**するロジックを実装します。

## 仕様の明確化事項

### 1. ポイント計算式の修正（v0.1からの修正）

ルール上、ポイント計算は以下の通り：

```
最終ポイント = base_point + 手札枚数 + 効果による加減算
```

v0.1 の `compare_points` は `base_point` のみで比較していたが、これはルールとの乖離。v0.2 で効果を導入するタイミングに合わせ、正しいポイント計算式に修正する。

> 注意: 瑞希 (buff_dynamic) の「手札枚数×1」は、基本ルールの「手札枚数ボーナス」とは**別の追加効果**として扱う。
> つまり瑞希のポイント = `base_point(17) + 手札枚数(基本ルール) + 手札枚数×1(効果分)`

### 2. ロコと貴音の `reveal` は挙動が異なる

`cards.jsonl` 上ではどちらも `"type": "reveal"` だが、実際の挙動は異なる：

- **ロコ (card_25)**: 相手の手札を**1枚ランダムに**選び、そのカードに**永続的な公開フラグ**を付ける（以降の全ラウンドでUIに表示し続ける）
- **貴音 (card_08)**: 相手の手札を**全て一時的に確認**する（そのラウンドの効果処理中のみ閲覧可能）

実装上は `cards.jsonl` 上の貴音の type を `reveal_all` に変更して区別する。

### 3. 百合子 (choose) の2段階インタラクション

百合子の効果は2段階の入力を要する場合がある：

1. **ステップ1**: 効果A（+3）か効果B（ドロー＆戻し）かを選択
2. **ステップ2**: 効果Bを選んだ場合 → 山札から2枚引いた後、**手札全体から2枚を選んで**山札の一番下に戻す

`INTERACTIVE_EFFECT` フェーズに**ステップ番号**の概念を持たせ、複数ステップに渡るインタラクションに対応する。

### 4. 亜美 (restart) の連続使用不可

亜美の効果「この効果は連続で使用することはできない」を管理するために、`GameState` に `last_restart_round: int | None` フィールドを追加する。直前のラウンドで restart が発動済みなら、亜美の効果は不発とする。

### 5. 千早 (negate) の処理順序

効果解決順序は「ポイントが小さい順」。千早 (basePoint=13) は多くのカードより先に処理される。

negate の実装方針：
- 千早の効果が処理された時点で、相手のカードに**効果無効フラグ**を立てる
- 以降、相手のカードの効果処理がスキップされる
- 千早より先にポイントが小さいカードの効果は既に処理済みとなるため、negateは遡及しない（先に処理された効果は有効のまま）

### 6. 真 (swap) 後のポイント比較と再判定

真の swap で場のカードが入れ替わった場合：
- **じゃんけん判定はやり直さない**（効果はあいこの後に発動するため、じゃんけんの段階は終了済み）
- **ポイント比較は入れ替え後のカードの base_point（+ 手札枚数 + 効果加減算）で行う**
- 場に新たに出たカードの戦具効果は発動しない（カードのテキスト通り）

### 7. ジュリア (removal) のラウンド消費ルール

ジュリアの効果が発動した場合：
- ジュリア自身は山札に戻る（プレイヤーの場が空になる）
- 相手の場のカードは捨札に送られる（**誰の勝ち札にもならない**）
- **ラウンドは消費される**（4ラウンド中の1つを使ったことになる）
- 勝敗判定はスキップされ、即座に次のラウンドへ進む

### 8. 杏奈 (copy_hand) のネスト効果

杏奈の効果で選んだ手札カードの効果がインタラクティブ（例：百合子）だった場合、さらに追加の入力が必要になる（ネスト発動）。

実装方針：
- `INTERACTIVE_EFFECT` フェーズに**効果処理スタック**の概念を導入
- 杏奈の効果で百合子を選んだ場合：杏奈の選択 → 百合子の選択（ステップ1 → ステップ2）と順番に処理する
- 13枚ではネスト効果の組み合わせが限定的（杏奈→百合子、杏奈→千鶴、杏奈→真、杏奈→ジュリア）

### 9. 未来 (copy_effect) のガード

13枚では各カードが1枚ずつ且つプレイヤーとNPCが別々のデッキを使うため、未来自身が山札の一番上に来ることはあり得る。

実装方針：
- めくったカードが `copy_effect` だった場合は**不発**（効果なし）として処理する
- めくったカードは山札の一番下に置く（不発でも同様）

- **レジストリパターン**:
  ```python
  EffectHandler = Callable[[GameState, Side, Card], GameState]
  _registry: dict[str, EffectHandler] = {}

  def register(effect_type: str):
      def decorator(fn: EffectHandler):
          _registry[effect_type] = fn
          return fn
      return decorator

  def resolve_effect(state: GameState, side: Side, card: Card) -> GameState:
      handler = _registry.get(card.effect.type.value)
      if handler is None:
          return state  # unknown effect = no-op
      return handler(state, side, card)
  ```
